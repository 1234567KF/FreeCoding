# 需求复杂度分级与前置准备指南

> 根据需求的预估工时和复杂度，分级定义前置准备的深度要求。与 [pre-spec-quality-scoring.md](pre-spec-quality-scoring.md) 联动，确保不同规模的需求获得匹配的前置投入。

## 一、复杂度分级标准

| 级别 | 名称 | 判定标准 | 典型场景 |
|------|------|---------|---------|
| L1 | 小微需求 | 预估工时 ≤4 小时 | Bug 修复、配置调整、单接口新增、UI 微调 |
| L2 | 中等需求 | 预估工时 4-16 小时 | 完整功能模块、多接口联调、页面新增 |
| L3 | 复杂需求 | 预估工时 ≥2 天 | 跨模块重构、新子系统、复杂业务流程、架构变更 |

**判定技巧**：如果无法确定工时，可通过以下问题辅助判定：

1. 涉及几个模块/服务？（1个→L1，2-3个→L2，≥4个→L3）
2. 需要修改数据库结构吗？（不需要→L1，新增表→L2，修改已有表结构→L3）
3. 需要多人协作吗？（不需要→L1/L2，需要→L2/L3）

## 二、分级前置准备要求

### L1 — 小微需求

| 项目 | 要求 |
|------|------|
| 前置准备时间 | ≥10 分钟 |
| 评分目标 | 均分 ≥3.5 |
| Q1-Q4 要求 | Q2（核心问题）和 Q3（实现范围）必须明确 |
| 额外准备物 | 无 |
| 典型 Spec 篇幅 | 1-2 页 |

### L2 — 中等需求

| 项目 | 要求 |
|------|------|
| 前置准备时间 | ≥30 分钟 |
| 评分目标 | 均分 ≥4.0 |
| Q1-Q4 要求 | 全部 4 个问题必须有清晰、具体的答案 |
| 额外准备物 | 以下至少提供 1 项：流程图初稿 / ER 图初稿 / 接口列表 |
| 典型 Spec 篇幅 | 3-5 页 |

### L3 — 复杂需求

| 项目 | 要求 |
|------|------|
| 前置准备时间 | ≥60 分钟 |
| 评分目标 | 均分 ≥4.5 |
| Q1-Q4 要求 | 全部 4 个问题必须有详细、无歧义的答案，含边界条件说明 |
| 额外准备物 | 必须提供以下全部：完整 ER 图 + 业务流程图 + 核心接口定义 + 风险清单 |
| 典型 Spec 篇幅 | 5-10 页 |

## 三、额外准备物规范

### 3.1 流程图 / ER 图

- 格式：Mermaid 语法、Draw.io 导出图、或手绘拍照均可
- 要求：核心实体/节点有明确命名，关系/流转方向清晰
- 粒度：L2 级别提供核心流程即可，L3 级别需覆盖主流程和主要分支

### 3.2 接口定义

- 格式：表格或 OpenAPI 片段
- 最低要求：接口路径 + HTTP 方法 + 核心请求/响应字段
- L3 级别额外要求：错误码定义、分页参数、权限要求

### 3.3 风险清单

- 格式：表格（编号 / 风险描述 / 影响程度 / 缓解措施）
- 最低要求：L3 级别至少识别 3 项风险
- 涵盖范围：技术风险、业务风险、进度风险

## 四、评分未达标的补救流程

当前置准备评分低于对应复杂度的目标分时：

| 情况 | 处理方式 |
|------|---------|
| 均分差距 ≤0.5 | 提示薄弱维度，用户确认后可继续进入 Step 2 |
| 均分差距 >0.5 且 ≤1.0 | 返回 Step 1，针对低分维度逐一补充，重新评分 |
| 均分差距 >1.0 | 返回 Step 1，建议用户先完成额外准备物再重新开始澄清 |

## 五、复杂度升级机制

在 Spec 编写过程中，如果发现实际复杂度超出初始判定：

1. Step 4 综合质量门禁未通过项 ≥3 个 → 考虑将复杂度上调一级
2. 复杂度上调后，按新级别的前置准备要求补充缺失的准备物
