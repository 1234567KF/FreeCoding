# 红队方案说明：kf-exa-code — Web-Scale Context Engine

## 核心创新：从"被动搜索"到"主动知识注入"

本方案的核心突破在于将 Exa 的 web-scale 检索能力从"用户主动查询"升级为 **编码过程中的隐式知识基础设施**。不同于现有搜索技能（kf-web-search 等）的被动响应模式，kf-exa-code 建立了三道渐进式防线：

### 激进创新点一：知识缺口自动检测

传统搜索技能等待用户/agent 明确提问。我们认为 LLM 编码时的最大效率损耗在于"模型不知道它不知道什么"——即模型在不确定的情况下生成低质量代码，而非主动寻求上下文。我们设计了一个连续运行的知识缺口检测器，通过扫描 import 语句、解析编译错误、识别 API 调用模式，在模型开始生成错误代码之前就自动触发检索。这从根本上改变了技能的使用模型：从 "search when asked" 变成 "search when needed"。

### 激进创新点二：投机预取（Speculative Pre-fetch）

这是最激进的性能优化。当 session 空闲时（等待用户输入），技能主动扫描项目依赖文件（package.json / pyproject.toml / Cargo.toml）和最近打开的文件，为所有依赖库预取代码示例并缓存。当用户最终需要时，知识已在缓存中，zero-latency 注入。这借鉴了 CPU 分支预测的思想——投机执行不浪费，但命中时收益巨大。保守方案会认为这是"浪费 token"，但我们算了一笔账：单次 Exa 搜索约 30 tokens，预取 10 个库共 300 tokens，而一次错误的代码生成可能需要 1000+ tokens 的调试和重写。

### 激进创新点三：代码片段融合（Code Snippet Fusion）

不满足于返回单个最佳示例，我们对多个高质量来源进行 AST 级别的融合——提取共有导入模式、选取最可读的实现、融入最佳错误处理，生成一个"融合片段"并标注每个部分的来源。这是对 Exa 原始"代码优先"哲学的激进延伸：不仅是返回代码，而是返回 **最优代码**。

### 激进创新点四：跨技能知识总线

我们设计了一个共享缓存层（`{IDE_ROOT}/exa-cache/`），所有 kf- 技能贡献和消费知识信号。当 kf-web-search 发现一篇好文章时，它写入信号文件，触发 kf-exa-code 刷新相关缓存。当 kf-spec 选定技术栈时，kf-exa-code 自动预取示例。这是对现有技能孤岛的彻底打破——"一次搜索，全技能受益"。

### 激进创新点五：反模式检测 + 许可证兼容性

在提供代码示例的同时，kf-exa-code 自动对比项目现有代码风格（从已有代码中检测命名规范、错误处理模式），在发现冲突时发出反模式警告，并提供适配后的代码变体。同时，通过读取包管理器文件检测项目许可证，自动判断检索到的代码示例的许可证兼容性——这在企业环境中具有实际的法律意义。

### 技术风险与缓解

- **预取可能浪费 token**：通过 TTL 缓存和 idle-only 执行，确保预取不影响主任务
- **Exa MCP 可能不可用**：自动降级到 kf-web-search 作为 fallback
- **代码融合可能出错**：保留原始来源链接，用户可追溯验证

### 评判维度预期

- **正确性 30%**：自动检测 + 多源验证 + 许可证检查，三重保障
- **效率 20%**：预取 + 缓存 + 融合，平均 latency 趋近于零
- **可维护性 20%**：文件系统缓存 + 清晰的分层架构
- **安全性 20%**：许可证检查 + 来源追溯 + 代码非执行原则
- **创新性 10%**：知识缺口检测 + 预取 + 融合 + 跨技能总线，四项原创
