# 绿队方案说明 — kf-exa-code 安全保守设计

## 整体思路

作为绿队（安全保守者），我们认识到 exa-code 的核心价值——通过 Exa MCP 搜索获取代码示例以解除 LLM 的知识瓶颈——但同时也清醒地看到其中的风险：外部 API 调用涉及数据泄露、服务不可用、速率限制、隐私合规等多重隐患。因此，我们的方案不是一个"尽可能多用 Exa"的设计，而是一个"在保证安全的前提下，审慎使用 Exa，并有完备后手"的设计。

## 六大安全保守设计要点

### 1. 安全前置层（Zero Trust Gateway）

设计理念是"默认不信任"——所有发往 Exa MCP 的请求必须经过三层安检：输入消毒（剥离业务代码，仅保留库名+API 签名）、隐私扫描（检测敏感信息模式）、速率检查（遵守 API 使用条款）。这是一个不可绕过的网关层，而非可选的建议。我们还在输出端增加了代码安全验证，在返回代码前检查可疑模式（eval、exec、shell 注入等），避免将安全风险带入项目。

### 2. 三级降级链（Graceful Degradation）

绿队设计的核心防御机制是三级降级链：
- **Level 0** (首选)：kf-exa-code 通过 Exa MCP 搜索
- **Level 1** (降级)：Exa 不可达/超时/限流时，自动降级到 kf-web-search
- **Level 2** (再降级)：Web 搜索不足时，降级到 kf-scrapling 深度抓取文档站点
- **Level 3** (兜底)：用户手动查阅文档

每级降级都是自动触发的，不需要用户手动操作。熔断器机制（连续 N 次失败后暂停一段时间）避免在服务异常时徒劳重试。

### 3. 全面错误处理（Defensive Programming）

我们覆盖了所有可预见的错误场景：MCP 连接失败、API 认证失败（401/403）、速率限制（429）、请求超时、空结果、低质量结果、敏感内容拦截。每种错误都有明确的识别方式、处理策略和用户通知策略。全局 try-catch 确保无未捕获异常，catch 后的默认行为是降级到 kf-web-search，而非崩溃或静默失败。

### 4. 隐私与合规（Privacy & Compliance）

最关键的设计决策：绝不将完整源文件或业务代码发送到 Exa。仅提取库名、API 签名、函数名等非敏感信息构成搜索查询。对于附带的代码片段，需经自动化清洗（变量名泛化、注释剥离）后才允许发送。如果用户明确要求发送敏感代码，绿队的立场是礼貌拒绝并建议使用本地知识库。遵守 Exa API 使用条款的速率限制（60 req/min），并声明 Exa 返回的代码示例仅作为开发参考使用。

### 5. 自动触发边界（Trigger Boundary）

我们明确定义了自动触发的边界条件：仅在编码场景涉及外部 API/库/SDK 时触发，不触发于纯架构讨论或业务逻辑实现。还定义了不触发条件（已在项目知识库中的 API 用法、用户明确不需要外部搜索等），避免不必要的外部请求。

### 6. 熔断与指数退避（Circuit Breaker + Backoff）

结合了熔断器和指数退避两种模式：单个请求失败时使用指数退避重试（1s→2s→4s），连续失败达到阈值时打开熔断器（30s→120s），熔断期间完全避免向 Exa 发送请求，直接使用降级方案。这既保护了 Exa 服务的稳定性，也避免了在错误状态下浪费 token 和用户时间。

## 与红蓝队的差异

- **与红队（创新激进者）**：红队可能追求最大化利用 Exa 能力，包括 deep 推理搜索、多轮迭代、更多场景自动触发。绿队则更关注边界条件和安全约束，宁可保守不可冒进。
- **与蓝队（务实平衡者）**：蓝队可能在安全与效率之间取折中，绿队则选择安全优先、降级完备的路线，增加了一定的设计复杂度但换取了更高的稳健性。

## 总结

绿队方案的核心理念是"安全不是特性，而是架构"。我们不是在已有功能上打安全补丁，而是从架构层面将安全、降级、熔断、合规作为一等公民设计进去。这使得 kf-exa-code 能够在生产环境中可靠运行，即使 Exa MCP 出现波动，也不会影响开发者的编码体验。
