# 蓝队方案说明 — kf-exa-code 技能设计

## 核心设计理念

作为蓝队（稳健工程师），我们遵循"能力边界清晰、降级路径明确、团队上手平滑"的设计原则。exa-code 的核心价值在于为 LLM 提供实时的代码上下文，但这个价值只有在稳定可靠的前提下才有意义 —— 一个经常超时或出错的技能，无论搜索多精准都不会被团队接受。

## 方案要点

### 1. 四层架构，各司其职

将技能拆为诊断层、MCP 客户端层、检索流水线层、集成层四层，每层有明确的输入输出接口。这样做有三个好处：一是新团队成员可以按层逐步理解，不需要一次消化全部逻辑；二是某层出 bug 时可以快速定位，不会牵一发动全身；三是后续要替换 MCP 客户端（比如 Exa 更新了协议版本）只需动一层，不影响上游流程。

### 2. 内建降级，不阻塞工作流

我们最担心的是：开发者正写到关键处，kf-exa-code 超时导致整个编码流程卡住。因此设计了四级降级方案（正常 -> 重试 -> kf-web-search -> 缓存兜底），确保上游技能（kf-spec、/夯）在任何情况下都不会被阻塞。降级时显式通知用户，但不停机。

### 3. 极简输出，尊重 token 预算

单结果控制在 200 token 内，整次调用控制在 500 token 内。我们做了两件事：一是把 Highlights 提取作为默认开启（10x token 效率），二是严格限制返回条数（1-3 条）。在输出模板中，代码块优先、说明极简、更多参考仅给链接。这样即使频繁调用，token 消耗也可预期。

### 4. 渐进式团队上手

新成员上手路径清晰：装环境 -> 跑 `/exa-code test` -> 试一次简单搜索 -> 理解输出格式。和现有技能（kf-web-search、kf-spec）的关系是互补而非替代，不增加团队认知负担。缓存机制也让频繁查询同一 API 的场景不再浪费 token。

### 5. 保守的集成策略

我们没有激进地让 kf-exa-code 自动注入到所有编码场景，而是设置了明确的触发条件（只有遇到未使用的第三方库 API 时才触发），并限制了调用次数（单 Agent 每阶段最多 5 次）。这样既解决了知识断层问题，又不会因为过度搜索打乱编码节奏。
