# 原型技能优化：kf-ui-prototype-generator

## 目标 (Goal)

确保 PRD 与原型保持一致，达到以下双重目的：

- **人类可读**：产品、开发、测试人员可直接从原型注释中核查业务逻辑，无需反复对照 PRD。
- **AI 可执行**：AI 可仅凭 PRD + 原型注释生成完整 Spec，在设计阶段不存在任何待决策的业务逻辑盲区。

> **关键区分**：本优化聚焦于业务逻辑表达的完整性，不涉及技术栈选型。技术栈选型不影响原型和 PRD 的呈现，二者应明确分离。

## 执行指令 (Action)

使用 `/kf-skill-design-expert` 对 `/kf-ui-prototype-generator` 进行优化或重构。

优化视角：**专业产品经理 + 前后端工程师**，确保注释内容覆盖业务语义与技术约束。

## 输入规格：原型注释必须包含的内容 (Input Spec)

### 1. 业务描述

每个页面/模块的注释须包含以下业务要素：

| 要素           | 说明示例                                         |
| -------------- | ------------------------------------------------ |
| 增删改查操作   | 明确该业务支持哪些操作                           |
| 状态流转       | 状态机定义：状态 → 触发条件 → 目标状态           |
| 数据权限       | 角色可见/可操作的数据范围                        |
| 数据校验       | 如：系统配置类数据，全系统仅允许一条记录         |
| 业务关联关系   | 与其他模块/实体的依赖与联动关系                  |

### 2. 查询字段

每个查询字段须标注：

| 属性         | 说明                       |
| ------------ | -------------------------- |
| 字段名       | 业务的字段标识             |
| 字段类型     | 数据类型（如 String/Int）  |
| 前端表现形式 | 输入框/下拉框/日期选择器等 |
| 数据校验规则 | 必填/格式/范围等约束       |

### 3. 列表字段

每个列表展示字段须标注：

| 属性         | 说明                       |
| ------------ | -------------------------- |
| 字段名       | 业务的字段标识             |
| 字段类型     | 数据类型                   |
| 字段说明     | 面向用户的语义说明         |
| 前端表现形式 | 文本/标签/徽章/链接等      |

### 4. 新增表单字段

每个新增表单字段须标注：

| 属性         | 说明                       |
| ------------ | -------------------------- |
| 字段名       | 业务的字段标识             |
| 字段类型     | 数据类型                   |
| 字段说明     | 面向用户的语义说明         |
| 前端表现形式 | 输入框/下拉框/开关等       |
| 数据校验规则 | 必填/格式/范围/边界值等约束 |

### 5. 修改表单字段

每个修改表单字段须标注（规格同新增表单，额外标注与新增的差异）：

| 属性             | 说明                                     |
| ---------------- | ---------------------------------------- |
| 字段名           | 业务的字段标识                           |
| 字段类型         | 数据类型                                 |
| 字段说明         | 面向用户的语义说明                       |
| 前端表现形式     | 输入框/下拉框/只读展示等                 |
| 数据校验规则     | 必填/格式/范围/边界值等约束              |
| 与新增表单的差异 | 如：某字段改为只读、默认值不同等         |

> **数据校验规则**须包含边界值定义（见下文）。

### 6. 异常处理与边界值表

| 条目         | 说明                                   |
| ------------ | -------------------------------------- |
| 异常场景描述 | 触发条件、错误信息、系统响应行为       |
| 边界值定义   | 最大输入长度、数值范围（min/max）等    |

## 输出规格 (Output Spec)

### 格式与模板

- 提供**标准注释模板**，确保每个页面的注释结构统一、字段信息无遗漏。
- 提供**完整示例**，演示模板的实际填充效果。
- 模板与示例须经过测试验证，确保输出稳定可复现。

### 样式与嵌入方式

- 注释样式须**贴合当前前端主题风格**，可无缝嵌入到原型页面或 MVP 实际前端代码中。
- 无论是原型产出还是 MVP 产出，均支持页面原型附加**暗门注释**（即开发者/审查者可见、终端用户不可见的注解层）。
- 注释信息须完整，不得因格式或空间限制而省略内容。

## 约束条件 (Constraints)

- **易读无歧义**：注释无论是文字、图标还是图片形式，均须语义明确，无二义性。
- **布局合理**：避免注释内容在狭长空间中展示，导致排版难看、不易阅读。
- **业务与技术分离**：注释内容聚焦业务逻辑与数据约束，不混入技术栈选型信息。

## 参考资料 (References)

- **kf-prd-generator** 技能中的 `SDD需求采集 - 实例2：政企用车需求撮合平台 SDD需求采集`，其需求采集方式具有参考价值。比如说状态机，ER图，不同颗粒度的业务流程表，对于可能涉及到的页面需要把这些内容展现出来。

基准是如果一个功能就是增删查改比如说企业信息管理，单纯的数据字段，没有其他太多关联依赖。那么他的注释就必然不多。而如果它有很多数据会影响其他功能作为判断的依据，那么写数据和读数据加载规则的地方都要把这个规则表格MECE无遗漏明确。甚至相关场景流程。
